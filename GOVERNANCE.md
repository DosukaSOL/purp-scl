# Governance

## Project Structure

Purp SCL is maintained by its core contributors. This document outlines how decisions are made and how the project is managed.

## Roles

### Maintainers
- Final say on releases, architecture decisions, and roadmap
- Review and merge pull requests
- Manage issues and project boards
- Enforce the Code of Conduct

### Contributors
- Submit pull requests, report issues, improve documentation
- Participate in discussions and RFCs
- Help other community members

### Community Members
- Use Purp, report bugs, suggest features
- Participate in discussions

## Decision Making

### Minor Changes
Bug fixes, documentation improvements, and small features can be merged by any maintainer after a single approving review.

### Major Changes
Significant features, breaking changes, or architectural decisions require:
1. An RFC (Request for Comments) as a GitHub Discussion or Issue
2. At least 7 days of community discussion
3. Approval from at least 2 maintainers
4. No unresolved blocking concerns

### Release Process
1. Feature freeze on `develop` branch
2. Release candidate testing
3. Changelog update
4. Version bump
5. Tag and publish
6. GitHub Release with notes

## Communication Channels

- **GitHub Issues**: Bug reports and feature requests
- **GitHub Discussions**: Community Q&A and RFCs
- **Pull Requests**: Code contributions and reviews

## Conflict Resolution

1. Discuss in the relevant GitHub issue or PR
2. If unresolved, maintainers will make a decision
3. Decisions can be revisited if new information emerges

## Amendments

This governance document can be amended through the standard Major Changes process described above.
